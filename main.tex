\begin{abstract}
A spanning tree of a graph is a connected, acyclic subgraph that includes all vertices. 
Enumerating all spanning trees of a graph is a fundamental problem in combinatorics 
with applications in network analysis, epidemiology, and circuit design. This report 
reviews the work of Liu, Wong, Lam, and Im (SODA 2026), which presents the first known 
pivot Gray code for spanning trees of complete graphs. A pivot Gray code is an ordering 
of all spanning trees such that consecutive trees differ by exactly one edge swap, where 
the removed and added edges share a common vertex. Their algorithm, GenS, generates all 
$n^{n-2}$ spanning trees of $K_n$ in pivot Gray code order in constant amortized time 
per tree using $O(n^2)$ space, resolving an open problem posed by Knuth rated 46 out of 
50 in difficulty. The framework also extends to edge-exchange Gray codes for general 
connected graphs.
\end{abstract}

\subsection{Description of Input}

The input is a graph $G = (V, E)$, which may be a complete graph $K_n$ or a general connected graph.

\subsection{Desired Output}

The output is a listing of all spanning trees of $G$, generated in Gray code order such that consecutive trees differ by a small edge modification (pivot or edge-exchange).

\subsection{Definitions and Preliminaries}

We introduce several key definitions used throughout this work.

\textbf{Spanning Tree.} A spanning tree of a graph $G$ is a connected, acyclic subgraph that includes all vertices of $G$.

\textbf{Gray Code.} A Gray code is an ordering of combinatorial objects such that consecutive elements differ by a small, local change. In this context, we consider Gray codes for spanning trees, where consecutive trees differ by an edge exchange.

\textbf{Edge-Exchange Gray Code.} An edge-exchange Gray code for spanning trees is a sequence in which each consecutive pair of trees differs by the removal of one edge and the addition of another.

\textbf{Pivot Gray Code.} A pivot Gray code is a stronger form of edge-exchange Gray code in which the removed and added edges share a common vertex. This imposes a stricter locality condition on transitions between consecutive trees.

\textbf{Amortized Time.} An algorithm is said to run in constant amortized time if the average time per operation over a sequence of operations is constant, even if some individual operations may take longer. In this work, we aim to generate each spanning tree in constant amortized time.

\textbf{Complete Graph.} A complete graph $K_n$ is a graph on $n$ vertices in which every pair of distinct vertices is connected by an edge.

\textbf{Mixed-Radix String.} A mixed-radix string is a sequence where each position may have a different base. This is used in the general graph case to represent valid parent assignments for vertices with differing adjacency constraints.


\section{Motivation}
\subsection{Real World Application}
\label{headings}
Spanning tree enumeration plays an important role across a wide range of scientific fields where alternative network configurations, robustness, and probabilistic behavior must be analyzed. 

One of the earliest motivations for spanning trees arises in 1847 from electrical network analysis. In electrical systems, Kirchhoff’s laws enable current and voltage analysis by decomposing a network into spanning trees \cite{kirchhoff1847}. Modern work further connects spanning trees to electrical flow optimization, where networks are analyzed through tree-based decompositions \cite{kelner2013electrical, borradaile2020electrical}. Power distribution systems are often operated as radial,tree-like, structures for stability and fault isolation, and enumerating spanning trees allows engineers to explore multiple configurations for power flow while also enabling optimization of energy efficiency, load balancing, and network reconfiguration.

In epidemiology, spanning trees provide a natural representation of transmission pathways, where each tree corresponds to a possible “who-infected-whom” structure during an outbreak. Transmission data is often incomplete or uncertain, so using enumerating spanning trees allows researchers to explore all plausible infection scenarios and perform probabilistic modeling of disease spread \cite{scott2021genomic, rahman2025epidemiology}. This enables better understanding of outbreak dynamics and supports evaluation of intervention strategies such as vaccination and contact tracing. In social network analysis, spanning trees are used to extract simplified backbone structures of complex interaction networks, enabling efficient sampling and analysis of connectivity patterns under structural constraints \cite{lee2024social}.

Spanning trees also play a key role in chemistry and biology by modeling molecular structures and evolutionary relationships. In phylogenetics, spanning trees represent candidate evolutionary trees, allowing researchers to compare competing hypotheses and quantify uncertainty in biological data \cite{graph_theory_mst_applications}. Spanning trees are used in biological sequence analysis, molecular modeling, and studies of network entropy. In physical and technological systems, spanning trees are closely related to models of diffusion, percolation, and connectivity in complex networks. They are also used in the design and analysis of telecommunication systems and in evaluating network complexity and resilience \cite{nature_complex_networks}.

In image processing and computer vision, spanning trees are applied to graph-based representations of images, where pixels are treated as nodes in a graph. Spanning trees enable image segmentation and clustering, grouping pixels into meaningful regions. Enumeration is particularly important in this setting, as different spanning trees correspond to different segmentations, allowing algorithms to explore multiple interpretations of an image and improve robustness \cite{gfg_spanning_tree,mst_image_segmentation}. Finally, in network design and reliability, spanning trees represent minimal connectivity structures in communication networks, power grids, and transportation systems. Enumerating spanning trees enables the evaluation of fault tolerance and reliability, since the number and diversity of spanning trees directly reflect the resilience of a network \cite{ding2024reliability}.

Across these domains, spanning tree enumeration supports three key objectives: reliability analysis, by quantifying alternative connectivity structures; probabilistic modeling, by exploring all feasible configurations; and optimization, by enabling selection among multiple candidate solutions. These applications highlight the practical importance of efficient spanning tree generation algorithms, such as those developed in this work.

\section{History}

\subsection{Counting Spanning Trees: Cayley's Formula}

The study of spanning trees began not in pure mathematics but in engineering. In 1902,
physicist Wilhelm Feussner investigated spanning tree enumeration in the context of
electrical network analysis, establishing a foundational question that researchers would
pursue for over a century \cite{liu2026}. The first major theoretical result was
Cayley's formula (1889): the complete graph $K_n$ has exactly $n^{n-2}$ spanning trees.
Pr\"{u}fer later formalized this with a bijection between spanning trees and integer
sequences, and Kirchhoff derived the same count independently through the matrix-tree
theorem \cite{liu2026}. To put the scale in perspective: $K_4$ has 16 spanning trees,
$K_5$ has 125, and $K_{10}$ has 100{,}000{,}000. This rapid growth is precisely what
makes efficient generation algorithms necessary.

\subsection{Early Algorithms and Their Limitations}

The first generation of algorithms, developed by researchers including Hakimi, Mayeda
and Seshu, Char, and Gabow and Myers, focused on producing every spanning tree exactly
once without regard for ordering \cite{liu2026}. While correct, these approaches had a
key limitation: consecutive trees in the output could differ by many edge changes
simultaneously. In practice, this makes the listings difficult to use in applications
such as network design, where a smooth, incremental transition between configurations
is often required.

\subsection{Gray Codes and the Revolving-Door Problem}

A more structured approach emerged from the study of Gray codes: orderings of
combinatorial objects where consecutive entries differ by a minimal, predictable change.
Frank Gray introduced the binary reflected Gray code (BRGC) in 1953, which lists all
$2^n$ binary strings of length $n$ so that each consecutive pair differs by exactly one
bit. For $n = 3$, this gives: 000, 001, 011, 010, 110, 111, 101, 100. The spanning
tree analog is an edge-exchange Gray code: a listing where each consecutive pair of
trees differs by removing one edge and adding one edge \cite{liu2026}.

Knuth posed the existence of such a listing for complete graphs as an open problem in
\textit{The Art of Computer Programming}, Volume 4 (Exercise 101, Section 7.2.1.6),
rating it 46 out of 50 in difficulty, and noted that existing approaches such as
Algorithm~S produce such listings but are ``quite complicated'' and lack a clean
structure. Cummins (1966) proved edge-exchange Gray codes exist for any connected
graph, with further proofs from Kamae, Kishi and Kajitani, and Holzmann and Harary.
However, all of these were existence results; none provided a practical construction
algorithm \cite{liu2026}.

\subsection{Constructive Algorithms}

Smith (1997) developed the first constructive algorithm for edge-exchange Gray codes
on spanning trees, producing a valid listing in constant amortized time per tree. This
means generating each new tree costs a fixed, bounded amount of work on average,
regardless of graph size. More recently, Merino, Mutze, and Williams achieved
$O(m \log n (\log \log n)^3)$ time per tree, later improved by Merino and Mutze to
$O(m \log n)$. Despite these advances, Smith's algorithm remains the only one achieving
constant amortized time \cite{liu2026}.

\subsection{The Pivot Gray Code}

Cameron, Grubb, and Sawada introduced a stricter variant: the pivot Gray code. Here,
the removed and added edges must share a common vertex, called the pivot. Not every
edge-exchange Gray code satisfies this; Smith's algorithm fails the pivot test for
$K_5$, where several transitions involve edges with no shared vertex. Cameron et al.\
developed a constant amortized time pivot Gray code algorithm for fan graphs, with
extensions to wheel graphs. Behrooznia and Mutze later extended this to outerplanar
graphs in $O(n \log n)$ time per tree. Both papers identified $K_n$ as the key
remaining open problem \cite{liu2026}.

\subsection{Algorithmic Challenges and Contributions}

Designing efficient algorithms for generating spanning trees in Gray code order presents
several challenges. First, the number of spanning trees grows exponentially as $n^{n-2}$,
making brute force enumeration infeasible for large graphs. Second, enforcing the Gray
code property requires that consecutive spanning trees differ by only a small local
modification, which is nontrivial to maintain while ensuring that all trees are generated
exactly once. A third challenge arises in imposing the stricter pivot property, where
the removed and added edges must share a common vertex.

Liu, Wong, Lam, and Im \cite{liu2026} resolve Knuth's open problem with the first known
pivot Gray code for spanning trees of any complete graph $K_n$. Their algorithm, GenS,
introduces a novel recursive framework that encodes spanning trees as tuples derived from
rooted tree representations, reducing the problem of generating spanning trees to
generating Gray codes over structured strings. For complete graphs, this is achieved
using $k$-ary Gray codes; for general graphs, the approach is extended using mixed-radix
Gray codes to account for varying adjacency constraints. By mapping changes in these
strings to modifications in the corresponding spanning trees, the algorithm ensures that
each step corresponds to a valid pivot operation.

GenS generates all $n^{n-2}$ spanning trees of $K_n$ in pivot Gray code order in
constant amortized time per tree using $O(n^2)$ space, matching the efficiency of
Smith's algorithm while satisfying the stricter pivot condition. Two key ideas make this
possible: first, Lemma 2.2 proves that every single-digit string change corresponds to
exactly one of four pivot change types on the tree, guaranteeing the pivot property is
preserved throughout the listing. Second, the reflectable languages framework used by
the GenK subroutine enables constant amortized time string generation. As a further
result, the paper derives a new combinatorial proof of Cayley's formula directly from
GenS's recursive structure, and extends the approach to edge-exchange Gray codes for
arbitrary connected graphs \cite{liu2026}.

\section{One Algorithm}
\label{others}

We elaborate on \textsc{GenS} (Algorithm 2.1 of Liu et al.~\cite{liu2026}), the procedure that generates a pivot Gray code listing of all $n^{n-2}$ spanning trees of the complete graph $K_n$. \textsc{GenS} is the headline result of the paper because it resolves Knuth's open problem from \textit{The Art of Computer Programming, Volume~4} (Exercise~101, Section~7.2.1.6), rated $46$ out of $50$ in difficulty. We pick \textsc{GenS} rather than its general-graph cousin \textsc{GenG} (Algorithm 3.2) because complete graphs admit a uniform encoding ($k$-ary strings instead of mixed-radix strings, a single connected component, and a one-dimensional recursion), which makes the main ideas easier to follow without sacrificing the central insight.

\subsection{About the Algorithm}

\paragraph{Encoding a spanning tree as a tuple.}
Every spanning tree of $K_n$ is rooted at vertex $1$. Once the root is fixed, organize the remaining vertices by their depth in this rooted tree, calling the depth the \emph{level} (root at level~$0$, its children at level~$1$, and so on). The whole spanning tree is then described by the tuple
\[
T = (t_1, t_2, \ldots, t_h),
\]
where $h$ is the height of the rooted tree and the $\ell$-th entry $t_\ell$ encodes which vertices land at level $\ell$ and who their parents at level $\ell-1$ are. Concretely, suppose levels $0$ through $\ell-1$ already contain $q_{\ell-1}$ vertices, with $p$ of them at level $\ell-1$. Then
\[
t_\ell = a_1 a_2 \cdots a_{n-q_{\ell-1}}
\]
is a string of length $n-q_{\ell-1}$ over the alphabet $\{0,1,\ldots,p\}$, called a $(p+1)$-ary string. The character $a_i$ describes the $i$-th smallest unplaced vertex: $a_i=0$ means that the vertex is not at level~$\ell$ at all, while $a_i=j>0$ means it is at level~$\ell$ and its parent is the $j$-th smallest vertex at level $\ell-1$. As an example from~\cite{liu2026}, the tuple $T=(00100,1100,22)$ encodes a particular spanning tree of $K_6$. The big idea of \textsc{GenS} is the following:
\begin{quote}
\emph{Listing all spanning trees of $K_n$ becomes the same task as listing all valid tuples~$T$.}
\end{quote}

\paragraph{The recursive idea.}
\textsc{GenS} starts from the path spanning tree $1{-}2{-}\cdots{-}n$, whose tuple is $T=(10^{n-2}, 10^{n-3}, \ldots, 1)$. It then recurses level by level. At level~$\ell$, with the subtree on levels $0$ through $\ell-1$ frozen, the algorithm sweeps through every legal value of $t_\ell$ in Gray code order using the helper \textsc{GenK} (Algorithm~2.2). For each value of $t_\ell$ produced, \textsc{GenS} recurses into level~$\ell+1$ to enumerate all spanning trees that share that prefix. When level~$\ell$'s string list is exhausted, control backtracks to level~$\ell-1$, which advances its own string and triggers a fresh sweep of the deeper levels.

\paragraph{Why this is a pivot Gray code.}
\textsc{GenK} produces a $1$-Gray code over $k$-ary strings: consecutive strings differ either by a single-symbol change or, in one special situation, by swapping a $0$ with a non-zero character. By Lemma~2.2 of~\cite{liu2026}, every such string change corresponds to exactly one of four edge edits on the spanning tree (illustrated in Figure~2.3 of the paper):
\begin{itemize}
    \item \textbf{Case (a)} $a_v\colon 0\to r>0$. A vertex $v$ at some level $q>\ell$ disconnects from its parent $w$ and reconnects to a new parent $u$ at level $\ell-1$. Removed edge $(v,w)$ and added edge $(v,u)$ both touch~$v$.
    \item \textbf{Case (b)} $a_v\colon r>0\to 0$. A vertex $v$ at level $\ell$ disconnects from its parent $u$ at level $\ell-1$ and re-attaches as a child of some other vertex $w$ at level $q\geq\ell$. Both edges touch~$v$.
    \item \textbf{Case (c)} $a_v\colon r\to j$ with $r,j>0$ and $r\neq j$. A vertex $v$ at level $\ell$ swaps its parent from $u$ to a different vertex $w$, both at level $\ell-1$. Both edges touch~$v$.
    \item \textbf{Case (d)} The $0$/non-zero swap. A vertex $u$ at level $\ell-1$ disconnects from its child $v$ at level $\ell$ and reconnects to a vertex $w\neq v$ at level $q>\ell$ (with $w$ a descendant of $v$, so the tree stays connected). Both edges touch~$u$.
\end{itemize}
In every case, the removed edge and the added edge share a common vertex, so consecutive trees differ by a \emph{pivot} edge exchange. This is the property that resolves Knuth's open problem.

\subsection{Pseudo Code}

The pseudocode of \textsc{GenS} is reproduced as Figure~\ref{fig:complete_algorithms}(a) (\texttt{GenS.png}). We annotate its lines below. The procedure receives a vertex set $V$ (the vertices not yet placed on levels $0$ through $\ell-2$) and the previous level's string $t_{\ell-1}$.
\begin{enumerate}
    \item \emph{Lines 2--3.} Split $V$ into $P$ (the vertices placed at level $\ell-1$, those with $a_i>0$ in $t_{\ell-1}$) and $C$ (the vertices still unassigned, those with $a_i=0$).
    \item \emph{Lines 4--6.} If $C$ is empty, the rooted tree is complete: print the spanning tree and return.
    \item \emph{Lines 7--9.} Initialize $t_\ell$ to all zeros, then mark in it any parent already chosen by the current spanning tree.
    \item \emph{Line 10.} Recurse: \textsc{GenS}$(C, t_\ell)$ enumerates all spanning trees with the current $t_\ell$ frozen.
    \item \emph{Lines 11--19.} Repeatedly call \textsc{NextK}$(t_\ell, |P|+1)$ to advance $t_\ell$ in $k$-ary Gray code order with $k=|P|+1$. The returned change set $\delta$ identifies which positions of $t_\ell$ moved, and the algorithm dispatches on its size:
    \begin{itemize}
        \item $|\delta|>1$: a $0$/non-zero swap, i.e.\ \emph{case (d)}. Reassign the parent pointer and call \textsc{Lift} to push the affected vertex up to level~$\ell$.
        \item $|\delta|=1$ and the new value $b_{\delta_0}>0$: \emph{cases (a) or (c)}. Set the parent of $c_{\delta_0}$ to the corresponding vertex in~$P$.
        \item $|\delta|=1$ and the new value is $0$: \emph{case (b)}. Re-route the parent of the affected vertex to a sibling already placed at level~$\ell$.
    \end{itemize}
    After each update, recurse into \textsc{GenS}$(C, t_\ell)$ again.
\end{enumerate}
The four edit cases of Lemma~2.2 thus correspond to the three branches of the dispatch in lines 13--19. The combined pseudocode of \textsc{GenS} and \textsc{GenK} fits comfortably within the one-page pseudocode budget specified for the report.

\subsection{Running Time}

\textsc{GenS} generates each spanning tree of $K_n$ in \emph{constant amortized time} using $O(n^2)$ space. The bound is established in Theorem~2.6 of~\cite{liu2026}.

\paragraph{Why \textsc{GenK} is constant amortized.}
\textsc{GenK} produces a $1$-Gray code over $k$-ary strings using the reflectable-language framework of Li and Sawada (cited as~[21] in~\cite{liu2026}). Its \textsc{NextK} subroutine produces the next $k$-ary string in the listing in constant amortized time by tracking only the position that changed since the previous call, so the per-string cost averages to $O(1)$.

\paragraph{Why \textsc{GenS} inherits the same bound.}
At each level $\ell$, the work performed before the while loop (lines 2--10) is $O(t)$ where $t = |t_{\ell-1}|$: we scan $V$ to build $P$ and $C$, and we copy known parent assignments into $t_\ell$. The while loop, in the worst case ($|C|=1$), produces $t-1$ different configurations of $t_\ell$, and in general, the number of configurations grows as the level descends. Therefore, the $O(t)$ setup cost is amortized over $\Omega(t)$ generated trees, giving a constant amortized cost per tree.

The \textsc{Lift} operation triggered on case (d) costs $O(t)$ in the worst case, but case (d) is invoked only once per stretch of $2^t$ updates of cases (a), (b), and (c), so its amortized cost is also constant. The remaining operations inside the while loop, namely updating a single parent pointer, looking up a vertex by index, and advancing $t_\ell$ via \textsc{NextK}, each take $O(1)$ when supported by the auxiliary hash tables described in Appendix~A of~\cite{liu2026}.

\paragraph{Space.}
\textsc{GenK}'s direction array $d$ stores $|t_\ell|$ entries at each of the $O(n)$ levels of recursion, contributing $O(n^2)$. The hash tables for the parent map $\pi$ and the sibling chain, together with the recursive stack for $T$, $P$, and $C$, also fit in $O(n^2)$. Hence, the total space is $O(n^2)$, matching Smith's earlier algorithm but achieving the strictly stronger pivot Gray code property.



\section{Implementation}

\subsection{Source and Setup}

The code implementation used in this project was provided from the research paper in Python, although the indentation of the code was not visible and needed to be fixed. The research paper provided 3 code snippets in Python. The first being code to generate pivot Gray codes for spanning trees of complete graphs. The second is to generate an edge-exchange Gray code for spanning trees of a general graph. The third is \texttt{k\_ary} that generates Gray codes for $k$-ary strings and mixed-radix strings. The other two programs utilize \texttt{k\_ary} to enumerate all spanning trees of the user-defined graph and print them out as a parent-pointer list \cite{liu2026}.

The program prompts the user to choose between five different graph types: complete graphs, fan graphs, wheel graphs, the Petersen graph, and custom graphs given by an edge list. The fan, wheel, and Petersen graphs are just different types of general graphs. For complete, fan, and wheel graphs, the user is prompted for the number of vertices within the graph. After input, the graph is constructed as an adjacency matrix, the program builds an initial spanning tree using breadth-first search, and then begins enumeration. Each spanning tree is outputted in the form of \texttt{1->-1; 2->1, 3->2; ...}, where every entry gives a vertex and the index of its parent, -1 representing the root \cite{liu2026}.

The programs only depend on Python's standard library and NumPy, which is utilized to build the adjacency matrix. No other packages are required, which keeps the setup rather minimalist and easy to implement in any Python standard environment. 

\subsection{Testing and Verification}

Testing is focused on two questions: is the correct number of spanning trees produced? Does the Gray code property hold across the output sequence? 

Correctness in regard to the count of spanning trees produced was verified using Cayley's formula that was previously mentioned, which states that a complete graph that has n vertices yields $n^{n-2}$ spanning trees. For n = 4 the expected count to see is 16, and for n = 5 it is 125. The counts with these corresponding n values, for a complete graph, are produced with the algorithm \cite{liu2026}. 

The Gray code property was verified by inspecting the output, by comparing consecutive lines. For each pair of consecutive trees, the parent-pointer strings were compared entry by entry. In each of the cases, exactly one or two entries differed between two consecutive trees. This is consistent with the pivot operation that was described earlier. In which one edge is removed and another is added, sharing a common vertex. In the program output, each spanning tree appeared only once. This confirms that the enumeration visits each spanning tree exactly once \cite{liu2026}. 

\subsection {Recommendations for Readers}

The following notes are intended for readers who wish to run or build on this implementation. 

\begin{itemize}
    \item Python 3.8 or later versions are required. The implementation utilizes \texttt{yield from} for recursive generator delegation. This is not available for earlier versions of Python.
    \item NumPy is necessary to be installed for all graphs, except complete ones. It is used to build the adjacency matrix for the graphs that need it. 
    \item Spanning tree counts grow rather rapidly as the number of vertices increases; this is more apparent for complete graphs. Cayley's formula gives a count of $n^{n-2}$ spanning trees, for complete graphs, which reaches a count of over a million at n = 9 and over one a billion at n = 11. Due to how large these numbers can be for such little n, it is suggested to test with smaller n values, as for larger n the printing process becomes time-consuming.
    \item For large graphs, the recursion depth can be quite substantial. When the recursion depth is too great, Python will raise the error \texttt{RecursionError}. If prompted with this error, one could raise the default recursion limit with \texttt{import sys; sys.setrecursionlimit(t)}, replacing \texttt{t} with any desired limit.
    \item In regard to custom graphs they must be input as a semicolon-separated list of edges and in the form of \texttt{1,2; 2,3; 3,4; 4,1}. The graph must be fully connected; if the graph is not, the BFS will fail and cause an infinite recursion loop, which in turn raises \texttt{RecursionError} due to the recursion depth.  
\end{itemize}


\section{Conclusion}
This report examined the work of Liu, Wong, Lam, and Im, which resolves a decades-long open problem by presenting the first known pivot Gray code for spanning trees of complete graphs. Their algorithm, GenS, encodes spanning trees as tuples of digit strings and guarantees the pivot property through four proven transition cases, achieving constant amortized time per tree using O(n²) space. Testing confirmed that the implementation produces correct tree counts matching Cayley's formula and valid pivot transitions throughout. The framework further generalizes to edge-exchange Gray codes for arbitrary connected graphs, extending the practical reach of the approach across applications in network design, epidemiology, and circuit analysis.
